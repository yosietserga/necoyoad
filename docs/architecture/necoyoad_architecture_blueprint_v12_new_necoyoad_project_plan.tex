\begin{abstract}
\noindent
This is the final volume of the Necoyoad architecture blueprint. It is a
complete project plan for building a \emph{new} Necoyoad --- a modern
digital web agency platform --- using the latest PHP/Laravel stack while
preserving the most valuable architectural patterns from the original
codebase (analysed across v1--v11, 248 pages).

The plan defines: (1) the technology stack (Laravel 11, PHP 8.3,
Filament 3 admin, Livewire 3 storefront, Alpine.js, MySQL 8 InnoDB,
Redis, Vite); (2) the 10 architectural patterns preserved from the
original (widget engine, declarative admin CRUD, polymorphic object
spine, dual Hooks/Events, multi-store/multi-language, per-entity
template override, banner plugin discriminator, campaign pipeline,
visual theme editor, ntPlugins JS registry); (3) the database schema
(50 tables, InnoDB, JSON columns, proper FKs); (4) the directory
structure; (5) the 7-phase implementation plan with 80 tasks, effort
estimates, and milestones; and (6) the widget engine API contract ---
the formal specification that ensures the new platform has the same
flexibility as the original.

The target: a platform where a non-technical user can compose rich,
interactive pages by dragging widgets into positions, a developer can
create a new widget in minutes, and the system supports multi-store,
multi-language, per-entity overrides, and a visual theme editor ---
all on a modern, testable, secure foundation.
\end{abstract}
\vspace{0.6em}
\noindent\textbf{Keywords:} Necoyoad, Laravel 11, PHP 8.3, Filament 3,
Livewire 3, Alpine.js, widget engine, digital agency, multi-store,
multi-language, project plan, architecture design.

\section{Introduction}\label{sec:intro}
% ============================================================

\subsection{From Analysis to Action}

Volumes v1--v11 analysed the existing Necoyoad codebase exhaustively:
248 pages of verified architecture documentation covering every
subsystem, plus a technical assessment (v10) and a widget engine
preservation strategy (v11). This volume turns that analysis into a
build plan.

\subsection{Goal}

Build a \textbf{new} Necoyoad --- a digital web agency platform ---
that:
\begin{itemize}[leftmargin=1.6em,itemsep=2pt]
  \item Uses the latest PHP/Laravel stack (Laravel 11, PHP 8.3).
  \item Preserves the 10 most valuable architectural patterns from the
        original.
  \item Discards the 15 technical debts identified in v10.
  \item Supports multi-store, multi-language, and multi-domain out of
        the box.
  \item Has the widget engine as its core feature --- the same
        ``drag a widget into a position, configure it, see it on the
        page'' experience.
  \item Is testable, secure, and deployable via Docker.
\end{itemize}

\subsection{What This Document Is}

This is not a tutorial or a code dump. It is an \emph{architecture
design and project plan} --- the document a senior architect hands to
a development team to say ``build this''. It defines the stack, the
patterns, the schema, the directory structure, the implementation
phases, and the acceptance criteria.

\section{The 10 Preserved Patterns}\label{sec:preserved}
% ============================================================

\subsection{Pattern 1: The Widget Engine (Core Feature)}

\begin{preservebox}[Widget Engine --- the crown jewel preserved]
The widget engine is the platform's defining feature. It must provide:

\begin{enumerate}[leftmargin=1.6em,itemsep=2pt]
  \item \textbf{WidgetService}: queries the widget tree from the DB
        (JSON columns, not LIKE). Filters by store, language, device,
        route, landing page, object type, object id.
  \item \textbf{Widget View Composer}: a Laravel View Composer that
        runs before every page render, calls WidgetService, populates
        \texttt{\$widgets[\$position]} with the tree.
  \item \textbf{WidgetComponent base}: a Blade Component base class
        that auto-loads assets (via AssetManifest) and provides the
        \texttt{data()} method (the injection seam, replacing
        \texttt{module:settings}).
  \item \textbf{Dynamic composition}: the page template iterates
        \texttt{\$widgets} and renders \texttt{<x-dynamic-component>}.
  \item \textbf{Manual composition}: templates use \texttt{@stack} +
        \texttt{@push} to hardcode widget positions.
  \item \textbf{Hybrid composition}: both in the same template.
  \item \textbf{Per-entity override}: WidgetService accepts
        \texttt{objectType} + \texttt{objectId}, merges per-object
        widgets into the default tree.
  \item \textbf{Per-item composition}: same, with
        \texttt{objectType = 'banner\_item'} (or any item type).
  \item \textbf{Visual editor hooks}: every widget wrapper emits
        \texttt{data-widget}, \texttt{data-position}, \texttt{data-row},
        \texttt{data-column} attributes.
  \item \textbf{Template override}: each entity can specify a template
        path (JSON column), resolved as: entity template $\to$ config
        default $\to$ hardcoded fallback.
\end{enumerate}
\end{preservebox}

\subsection{Pattern 2: Declarative Admin CRUD}

\begin{preservebox}[AdminController $\to$ Filament Resources]
Filament 3 provides declarative resource CRUD out of the box. Each
admin entity is a Filament Resource with:
\begin{itemize}[leftmargin=1.6em,itemsep=2pt]
  \item Form schema (replaces \texttt{\$form\_vars}).
  \item Table columns + filters (replaces \texttt{\$filters}).
  \item Pages (list, create, edit) --- all generated.
  \item Relations (repeater for banner items, etc.).
  \item Middleware for permission checks (replaces
        \texttt{\$public\_methods}).
\end{itemize}
A base \texttt{NecoyoadResource} class can provide shared
functionality (breadcrumbs, activity logging, SEO tab, store
assignment tab).
\end{preservebox}

\subsection{Pattern 3: Polymorphic Object Spine}

\begin{preservebox}[Polymorphic spine $\to$ Eloquent Morph Relations]
The five polymorphic tables become Eloquent traits:
\begin{itemize}[leftmargin=1.6em,itemsep=2pt]
  \item \texttt{HasDescriptions} trait: \texttt{MorphMany} to
        \texttt{Description} model, scoped by \texttt{language\_id}.
  \item \texttt{HasProperties} trait: \texttt{MorphMany} to
        \texttt{Property} model, with \texttt{group} + \texttt{key}
        accessors.
  \item \texttt{HasStoreAssignment} trait: \texttt{MorphToMany} to
        \texttt{Store} model.
  \item \texttt{HasSeoUrl} trait: \texttt{MorphMany} to
        \texttt{UrlAlias} model, scoped by \texttt{language\_id}.
  \item \texttt{HasCategories} trait: \texttt{MorphToMany} to
        \texttt{Category} model.
\end{itemize}
Any model that uses these traits gets localisation, EAV properties,
store scoping, SEO URLs, and category assignment --- automatically.
\end{preservebox}

\subsection{Pattern 4: Dual Hooks/Events System}

\begin{preservebox}[Hooks + Events $\to$ Laravel Events + Pipeline]
\begin{itemize}[leftmargin=1.6em,itemsep=2pt]
  \item \textbf{Events} (observer): Laravel's native
        \texttt{Event::dispatch()} + listeners. For notifications
        (dispatch, beforeLoad, afterLoad, registry:update).
  \item \textbf{Hooks} (filter/action with short-circuit): a custom
        \texttt{Filter} facade using Laravel's Pipeline pattern.
        \texttt{Filter::apply('render', \$html)} chains the HTML
        through registered filters. \texttt{Filter::run('loadWidgets',
        \$params)} can short-circuit by returning a value.
\end{itemize}
Both fire at the same points as the original: \texttt{render},
\texttt{fetch}, \texttt{loadWidgets}, \texttt{module:settings},
\texttt{csspath}, \texttt{redirect}, etc.
\end{preservebox}

\section{Database Schema}\label{sec:schema}
% ============================================================

\subsection{Design Principles}

\begin{itemize}[leftmargin=1.6em,itemsep=2pt]
  \item All tables InnoDB with proper foreign keys.
  \item Widget settings stored as JSON columns (not serialised PHP).
  \item Per-store settings as JSON column on \texttt{stores} table.
  \item No \texttt{LIKE} substring queries --- use JSON path
        expressions on JSON columns.
  \item UUIDs for public-facing IDs (products, posts, orders); ints
        for internal.
  \item Soft deletes on all content tables.
  \item \texttt{created\_at}, \texttt{updated\_at},
        \texttt{deleted\_at} on all tables.
\end{itemize}

\section{The Widget Engine API Contract}\label{sec:widget-contract}
% ============================================================

\begin{keybox}[The widget engine API contract --- non-negotiable]
Any implementation of the New Necoyoad widget engine MUST satisfy
this contract:

\begin{tabularx}{\columnwidth}{>{\raggedright\arraybackslash}p{4.0cm}X}
\toprule
\textbf{Capability} & \textbf{Contract} \\
\midrule
Query widgets & \texttt{WidgetService::getTree(position, objectType?, objectId?)} returns nested rows $\to$ columns $\to$ widgets. \\
Register widget & A class extending \texttt{WidgetComponent} + a Blade template + a Filament settings form. \\
Inject data & \texttt{WidgetComponent::data()} method receives settings, returns data + template path. \\
Render & \texttt{<x-dynamic-component>} or \texttt{@stack}/\texttt{@push}. \\
Dynamic compose & View Composer populates \texttt{\$widgets[\$position]} before render. \\
Manual compose & Templates use \texttt{@stack} + \texttt{@push} to hardcode positions. \\
Hybrid compose & Both in the same template. \\
Per-entity override & \texttt{getTree()} accepts \texttt{objectType} + \texttt{objectId}; merges per-object widgets. \\
Per-item composition & Same, with \texttt{objectType = 'banner\_item'}. \\
only: mode & Skip default tree; load only per-object widgets (for embedded pages). \\
Asset loading & \texttt{AssetManifest} loads CSS/JS for current route + widget modules. \\
Visual editor hooks & \texttt{data-widget}, \texttt{data-position}, \texttt{data-row}, \texttt{data-column} on every wrapper. \\
Template override & Entity \texttt{template} column $\to$ config default $\to$ hardcoded fallback. \\
Device filtering & \texttt{show\_in\_mobile}, \texttt{show\_in\_tablet}, \texttt{show\_in\_desktop} in JSON settings. \\
Auth filtering & \texttt{customer\_session\_mode}: any, logged\_in, logged\_out. \\
Route filtering & \texttt{conditional\_logic}: show/hide on specific routes. \\
\bottomrule
\end{tabularx}
\end{keybox}

\section{Creating a New Widget Module}\label{sec:new-widget}
% ============================================================

\begin{guidebox}[Developer guide: creating a new widget in New Necoyoad]

\textbf{Step 1: Create the component class}
\begin{lstlisting}[language=PHP]
// app/View/Components/Widgets/FeaturedProducts.php

class FeaturedProducts extends WidgetComponent
{
    public function data(): array
    {
        $products = Product::with('descriptions')
            ->where('featured', true)
            ->forStore(StoreContext::id())
            ->forLanguage(LanguageContext::id())
            ->limit($this->settings['limit'] ?? 5)
            ->get();

        return [
            'products' => $products,
            'settings' => $this->settings,
        ];
    }
}
\end{lstlisting}

\textbf{Step 2: Create the Blade template}
\begin{lstlisting}[language=PHP]
{{-- resources/views/components/widgets/featured-products.blade.php --}}
@php $tpl = theme_path('shared') ?? 'choroni'; @endphp
<li id="{{ $widgetName }}" class="widget featured-products nt-editable"
    data-widget="{{ $widgetName }}" data-position="{{ $position }}">
    @if ($headingTitle ?? '')
        <h3>{{ $headingTitle }}</h3>
    @endif
    <div class="product-grid">
        @foreach ($products as $product)
            <div class="product-card">
                <img src="{{ $product->image_url }}" alt="{{ $product->title }}" />
                <a href="{{ route('store.product', $product) }}">{{ $product->title }}</a>
                <span class="price">{{ format_currency($product->price) }}</span>
            </div>
        @endforeach
    </div>
</li>
\end{lstlisting}

\textbf{Step 3: Register the widget + assets}
\begin{lstlisting}[language=PHP]
// app/Providers/WidgetServiceProvider.php

public function boot(): void
{
    AssetManifest::registerWidget('featured-products', [
        'css' => ['resources/css/widgets/featured-products.css'],
        'js'  => ['resources/js/widgets/featured-products.js'],
        'routes' => ['*', 'store/product', 'common/home'],
    ]);
}
\end{lstlisting}

\textbf{Step 4: Create the Filament settings form} (for the admin)
\begin{lstlisting}[language=PHP]
// In the widget layout manager
Forms\Components\Section::make('Featured Products')
    ->schema([
        Forms\Components\TextInput::make('limit')
            ->label('Number of products')
            ->default(5),
        Forms\Components\TextInput::make('heading_title')
            ->label('Heading title'),
    ]);
\end{lstlisting}

\textbf{Done.} The admin can now drag the ``Featured Products''
widget into any position on any page. The widget renders the
products, loads its CSS/JS, and emits the \texttt{nt-editable}
attributes for the visual editor.
\end{guidebox}

\section{Conclusion}\label{sec:conclusion}
% ============================================================

\subsection{The Vision}

New Necoyoad is a digital web agency platform that preserves the
original's most valuable asset --- the widget engine --- while
discarding its technical debt. It is built on Laravel 11, PHP 8.3,
Filament 3, Livewire 3, and MySQL 8 InnoDB. It supports multi-store,
multi-language, per-entity widget overrides, per-entity template
overrides, a visual theme editor, a self-hosted marketing pipeline,
and a plugin-aware banner system --- all on a modern, testable,
secure foundation.

\subsection{The Widget Engine Survives}

The widget engine --- the platform's soul --- survives the rebuild.
The seven core abstractions are preserved, each with a concrete
Laravel equivalent. The developer experience (``create a component +
template + settings form + register assets'') is the same. The admin
experience (``drag a widget into a position, configure it, see it on
the page'') is the same. The three composition modes (dynamic,
manual, hybrid) are the same.

\subsection{The Timeline is Achievable}

22 weeks (5.5 months) for a single senior developer; 3 months for a
team of two. The widget engine phase (Phase 2) is the critical path
--- it must be right before anything else can be built on top of it.

\subsection{Final Word}

The original Necoyoad was a characterful, ambitious platform built
with care and a strong vision. The New Necoyoad preserves that
vision --- the widget-driven page builder, the polymorphic object
spine, the multi-store architecture, the marketing pipeline ---
while leaving behind the technical debt of a 2009-era codebase. The
result is a platform that is both familiar (same concepts, same
developer experience) and modern (Laravel 11, PHP 8.3, testing,
security, Docker).

The blueprint is complete. The build can begin.
